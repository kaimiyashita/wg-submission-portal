## 2. データ設計(確定・版1.0)

### 2.1 リスト「成果物リスト」 ─ 1行 = 1アプリ

| 表示名 | 内部名 | 型 | 備考 |
|---|---|---|---|
| アプリ名 | Title | 1行テキスト | 既定列を流用 |
| 氏名 | Shimei | 1行テキスト | 初回提出時に確定、**編集不可** |
| アプリ概要 | Gaiyo | 複数行テキスト(書式なし) | 必須/後日提出可 |
| プロンプトのレシピ | Recipe | 複数行テキスト(書式なし) | 必須/後日提出可 |
| フォルダURL | FolderUrl | 1行テキスト | 提出フロー内で書き戻し |

- ID(自動採番)・作成者(Author)・作成日時(Created)は既定列を利用
- ✅ 内部名は実環境(テスト用リスト「成果物リスト_test」)で作成・実測済み。上表の内部名(Title/Shimei/Gaiyo/Recipe/FolderUrl)は表示名と一致することを確認済み
- リスト詳細設定「作成されたアイテムのみ編集を許可」を **ON**(編集ガードの本体)
- アイテム作成/更新のREST呼び出し時、リクエストボディの`__metadata.type`はテスト環境で `SP.Data.成果物リスト_testListItem` (本番リスト「成果物リスト」作成時は同様のパターンで`SP.Data.成果物リストListItem`になる想定だが、こちらは未実測のため本番リスト作成後に要確認)

### 2.2 ドキュメントライブラリ `成果物提出/`

- フォルダ命名: **`氏名_{ID3桁ゼロ埋め}`**(例: `山田太郎_003`)。ID部がユニーク性を保証、氏名部は可読性用。作成後は不変(リネーム処理は作らない)
- フォルダ内ファイルは**固定名に強制リネーム**して保存:
  - `source.zip`(ソースコード・必須/後日提出可)
  - `report.md`(ライトなレポート・必須/後日提出可)
  - `slides.pptx`(発表資料・必須/後日提出可)
- 3ファイルとも最終的には提出必須だが、初回提出時点で揃っている必要はなく、未提出のまま送信してよい(不足分は一覧のステータスバッジで示し、編集画面から後日追加する)
- 氏名・アプリ名の記号(`\ / : * ? " < > | # % & { } ~`)はフォーム側で自動除去

## 3. 画面仕様(確定・版1.0)

SPA構成。ハッシュルーティング: `#/`(一覧) `#/submit`(提出) `#/app/{ID}`(詳細) `#/edit/{ID}`(編集)。

### 3.1 提出フォーム(#/submit)

- 氏名: `/_api/web/currentuser` から自動取得(編集可)。取得失敗時は手入力を促す
- **必須(送信をブロックする)のは氏名とアプリ名の2項目のみ**。「必須」表記もこの2項目にだけ付与する
- アプリ概要・プロンプトのレシピ・ファイル(zip/md/pptx)は入力・提出がなくても送信をブロックしない(未入力/未提出のまま送信し、後日編集画面から追記・追加可)
- **フォーム上部(タイトル直下)に案内文を1つ表示**し、「後で追記・追加してよい」旨はここに集約する(各項目には繰り返し書かない):
  - 提出画面: 「必須は氏名とアプリ名だけです。他の項目とファイルは、後日このポータルの編集画面からいつでも追記・追加できます。まずは登録だけでもOKです。」
  - 編集画面: 「ファイルを差し替えると元のファイルは復元できません」
- 各項目の直下の注意書きは「何を書くか」の内容ガイドに専念させる(該当項目のみ、記載のない項目は注意書きなし):
  - 氏名(提出・編集共通): 「ログインユーザーから自動入力されます。初回提出後は変更できません。」
  - アプリ概要: 「どんなことができるアプリかを書いてください。READMEの流用OK。」
  - プロンプトのレシピ: 「「これは使えた!」というお気に入りプロンプトの具体例を書いてください。」
  - ソースコード(zip): 「APIキー等のシークレットは必ず除いてください。上限50MB。」(上限50MBの記載はzipの説明欄にのみ記載し、他の項目やラベルには繰り返さない)
  - レポート(md): 「開発を通じて発見したこと・気づき・理解したことをまとめたライトなレポート(Markdown形式)。詳細画面でプレビュー表示されます。」
  - 発表資料(pptx): 「レポートをサクッとまとめたスライド(3〜4ページ想定)。」
- ファイルは上限50MB/ファイル(zip/md/pptx共通)、超過時はエラー表示: 「ファイルが大きすぎます(上限50MB)。node_modules等の依存フォルダが含まれていないか確認してください。」
- ファイルは拡張子を固定(zip→`.zip`、md→`.md`、pptx→`.pptx`)で制限する。`<input type="file">`の`accept`属性で選択時に絞り込みつつ、送信時にも拡張子を検証し、異なる拡張子の場合は「.zip形式のファイルを選択してください」のように送信をブロックする(`accept`だけでは「すべてのファイル」を選べば回避できるため、送信時チェックが実質的なガード)
- 送信後: 完了画面+「続けて別のアプリを提出」「一覧を見る」導線(1提出=1アプリ。複数アプリは繰り返し提出)

### 3.2 提出一覧(#/)

- 上部: 提出期限表示+サマリー(提出アプリ数 / 提出者数)
- カードグリッド。各カード:
  - **サムネイル**: `slides.pptx` を `/_layouts/15/getpreview.ashx?path={ファイルパス}` で画像取得。未提出時はプレースホルダー(イニシャル等の生成デザイン)
  - アプリ名 / 氏名 / 概要(4行で省略、未登録時は「(アプリ概要は未登録です)」と表示)
  - **ステータスバッジ(自動判定)**: 3点(zip/md/pptx)すべて揃い→「提出完了」、不足がある場合は不足しているものを具体的に表示(例:「スライド待ち」「ソース・レポート待ち」)。列は持たず、フォルダ内ファイル一覧APIの結果から判定
  - 自分のアイテム(Author = currentuser)にのみ「編集」ボタン表示
- 並び: 新着順。v1ではフィルターなし

### 3.3 詳細ページ(#/app/{ID})

URLをTeamsに貼って共有できることが要件。上から順に:

1. ヘッダー(アプリ名・氏名・ステータス)
2. アプリ概要(未登録時は「(アプリ概要は未登録です)」と表示)
3. **プロンプトのレシピ**(全文。ナレッジ共有の主役なので上位配置。未登録時は「(レシピの登録はありません)」と表示)
4. **レポート**: `report.md` を `GetFileByServerRelativeUrl('...')/$value` で取得しMarkdownレンダリング(サニタイズ必須)
5. **スライド**: Office Online(WOPI)埋め込みiframeでその場プレビュー。埋め込み不可の環境ではフォールバックとして「ビューアーで開く」リンク
6. 提出フォルダへのリンク(zipはここから)

アプリ実体(zipの中身)の実行プレビューは**スコープ外**(将来拡張の余地としてのみ記載)。

### 3.4 編集(#/edit/{ID})

- 提出フォームを再利用し既存値をプリフィル。**氏名は変更不可**
- ファイル3点は各々「変更しない / 差し替える」の選択式。差し替えは同名 `overwrite=true` 上書き
- 本人判定はAuthor基準(UI出し分け)+リスト設定「作成したアイテムのみ編集」(サーバー側ガード)
- 削除UIは作らない(取り下げは運営がリスト側で対応)

## 4. 提出フロー(確定・版1.0)

```
1. リストにアイテム作成 → 応答からID取得
2. フォルダ `氏名_{ID3桁}` を作成
3. source.zip / report.md / slides.pptx をアップロード(固定名にリネーム)
4. アイテムに FolderUrl を書き戻し
```

- **冪等性要件**: 全操作は再実行しても安全であること。途中失敗時はエラー表示+同画面から再送信可能とし、再送信は既存アイテム/フォルダを再利用して上書きする(初回失敗のリカバリを編集と同じ経路に乗せる)
- エラー表示は生HTTPメッセージではなく**エラーコード+報告用文言+コピーボタン**の形式

### 使用するREST APIまとめ

| 用途 | エンドポイント |
|---|---|
| ダイジェスト取得 | `POST /_api/contextinfo` |
| ユーザー取得 | `GET /_api/web/currentuser` |
| アイテム作成/更新 | `POST /_api/web/lists/getbytitle('{list}')/items` |
| アイテム一覧 | `GET .../items?$select=...&$orderby=Created desc` |
| フォルダ作成 | `POST /_api/web/folders` |
| ファイルUP | `POST .../GetFolderByServerRelativeUrl('...')/Files/add(url='...',overwrite=true)` |
| ファイル取得 | `GET .../GetFileByServerRelativeUrl('...')/$value` |
| フォルダ内一覧 | `GET .../GetFolderByServerRelativeUrl('...')/Files`(ステータス判定用) |
| サムネイル | `GET /_layouts/15/getpreview.ashx?path=...` |

書き込み系は `X-RequestDigest` ヘッダー必須、全リクエスト `credentials: "same-origin"`。

## 6. 運用設計

### リリース・切り戻し
- ポータルHTML自体をバージョン管理有効のライブラリに配置。差し替え時はバージョンコメントに変更内容を1行記載。障害時は「以前のバージョンに復元」で即ロールバック
- コード内 `CONFIG.version` を更新し画面に表示(報告時の版特定用)

### バグ対応
- 受け口: WGチャネルに「ポータル不具合報告」スレッド(スクショ+操作内容+画面のエラーコード)
- 修正: ローカル(デモモード)で再現・修正 → テスト用リストで実環境確認 → 本番へ上書きアップロード

### 非常時の逃げ道(最初から周知)
- ポータル不使用でも提出可: ライブラリに直接フォルダを作りファイルを置いてチャネルで一報 → 運営がリストへ手動登録。データ本体はSharePoint標準機能上にあるため、ポータル全損でも提出は止まらない

### 締め切り後
- リストの編集許可をOFFにして凍結(コード変更不要)
- 発表会: 詳細ページURLを発表順に並べたリンク集で進行
- WG終了後: レシピをエクスポートし「プロンプトレシピ集」として全社共有(WG成果報告に活用)

### その他
- モバイル: 閲覧(一覧・詳細)のみレスポンシブ対応。提出はPC前提
- 対応ブラウザ: Edge / Chrome
